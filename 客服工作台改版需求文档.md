# CardBrother 客服工作台改版需求文档

版本：v0.1 草案  
背景：客服端当前主要是聊天页，已能切换客户、收发消息、备注、翻译、发起视频，但缺少围绕单个客户的业务资料和处理入口。  
目标：把“客服聊天页”升级为“客服工作台”，让客服能一边沟通，一边查看和处理客户订单、余额、提现、贷款、备注等关键业务信息。

## 1. 合理性检查结论

当前产品形态里，普通客户侧已经有较完整的业务页面：

- 首页：汇率、国家切换、卖卡入口。
- 卖卡页：填写卡片信息，生成订单并通知客服。
- 订单页：查看订单状态。
- 钱包页：查看可用余额和待结算金额。
- 提现页：提交提现资料。
- 贷款页：提交贷款申请。
- 排行榜、个人中心、视频通话等辅助功能。

客服侧目前集中在一个聊天页面，主要能力是：

- 客户会话列表。
- 收发消息。
- 未读红点。
- 备注客户。
- 用户消息自动翻译中文。
- 群发自己客户。
- 视频通话。
- 客户数量和余额总量汇总。

这个设计对于“只做在线聊天”是够的，但对于 CardBrother 这种卖卡、结算、提现、贷款都需要客服跟进的平台不够。客服不能只看消息，必须能看到“这个客户现在有什么业务要处理”。否则客服需要在聊天、订单、提现、贷款等页面之间跳转，效率低，也容易漏单、漏提现、漏风险。

结论：需要大改，但不是推倒重做。合理方案是在现有 `support-chat-v2` 的基础上扩展成三栏工作台，并补齐后端的“客户聚合详情接口”。

## 2. 客服主要作用

客服不是普通聊天对象，而是平台运营处理人员。核心职责应包括：

1. 接待客户：及时响应客户消息，理解客户卖卡、提现、贷款、视频沟通等诉求。
2. 处理卖卡订单：查看客户提交的订单资料、卡种、面额、数量、图片凭证、备注，推进状态。
3. 跟进结算余额：知道客户可用余额、待结算金额、已提现金额，解释结算进度。
4. 处理提现：查看客户提现资料，完成线下/人工打款后标记完成。
5. 协助贷款/风控：查看客户贷款申请，知道状态和审核备注；如客服无审核权，也要能解释进度。
6. 维护客户资料：给客户打备注，识别重点客户、风险客户、语言偏好、沟通历史。
7. 主动运营：对自己名下客户群发通知，跟进沉默客户或活动客户。
8. 升级问题：遇到争议订单、异常提现、贷款问题时，给管理员提供完整上下文。

所以客服工作台的第一原则是：以客户为中心，把“聊天 + 客户资料 + 业务处理”放在同一个操作界面里。

## 3. 改版目标

### 3.1 产品目标

1. 客服登录后进入统一工作台，而不是只进入单一聊天窗口。
2. 选择某个客户后，能同时看到聊天记录和该客户业务资料。
3. 客服能快速判断客户当前需要处理什么：未读消息、待处理订单、待处理提现、待处理贷款。
4. 客服能在一个页面完成常用动作：回复、备注、翻译、视频、群发、改订单状态、改提现状态。
5. 管理员后续可以复用类似结构，只是范围扩大为全部客户和全部客服。

### 3.2 技术目标

1. 复用现有聊天、订单、提现、贷款、余额、备注接口和权限规则。
2. 新增一个“客服客户详情聚合接口”，减少前端多次过滤和拼装。
3. 前端先改 `support-chat-v2` 为工作台布局，避免新增太多重复页面。
4. 保留移动端可用性：手机端可以用分屏/抽屉切换，桌面端用三栏。

## 4. 推荐页面结构

### 4.1 桌面端三栏

左栏：客户队列

- 搜索客户。
- 客户备注名/原用户名。
- 在线状态。
- 最近消息。
- 未读数。
- 最近更新时间。
- 快捷标签：待订单、待提现、待贷款、重点客户。

中栏：聊天区

- 当前客户聊天记录。
- 用户消息中文翻译。
- 图片/GIF/语音/文本。
- 粘贴图片待发送。
- 发送消息。
- 视频通话。
- 客户备注编辑。
- 群发入口可以保留在工具栏或独立按钮。

右栏：客户资料与业务面板

- 客户概览。
- 余额摘要。
- 订单列表。
- 提现列表。
- 贷款列表。
- 视频记录。
- 通知/事件时间线。
- 内部备注。

### 4.2 移动端分层

移动端不能强行三栏，建议：

- 默认显示客户列表。
- 点击客户进入聊天。
- 聊天页顶部增加“资料”按钮。
- 点击“资料”打开底部抽屉或右侧抽屉，展示客户业务面板。
- 订单、提现、贷款列表在抽屉内用 tabs 切换。

## 5. 客户详情面板字段

### 5.1 客户概览

必须展示：

- 客户备注名。
- 原用户名。
- 在线/离线。
- 当前专属客服。
- 注册时间。
- 最近登录/最近活跃时间，如果后端有数据则展示。
- 客户状态：正常、封禁、风险，先按已有用户状态展示。
- 联系方式：邮箱、手机号，如果当前模型有值。

建议展示：

- 客户国家/结算国家。
- 语言偏好。
- 累计订单数。
- 累计完成金额。
- 待处理事项数量。

### 5.2 余额摘要

展示：

- 可用余额。
- 待结算金额。
- 已提现金额。
- 订单完成金额。
- 待处理订单金额。

说明：当前后端已有 `BalanceSummary`，但它是范围汇总，不是单客户汇总。需要新增按客户计算的余额摘要。

### 5.3 订单列表

展示当前客户订单：

- 订单号。
- 卡种。
- 面额。
- 结算金额。
- 状态：pending、processing、completed、disputed。
- 图片凭证。
- 备注。
- 创建时间、更新时间。

客服可操作：

- pending -> processing。
- processing -> completed。
- pending/processing -> disputed。
- 点击订单跳转或在右栏展开详情。

注意：客服只能处理自己客户的订单，管理员可处理全部。

### 5.4 提现列表

展示当前客户提现：

- 提现单号。
- 金额。
- 国家。
- 收款人。
- 银行名称。
- 收款账号。
- 联系方式。
- 备注。
- 状态：pending、completed。
- 创建时间、更新时间。

客服可操作：

- pending -> completed。

注意：提现涉及资金，状态变更需要二次确认。

### 5.5 贷款列表

展示当前客户贷款：

- 申请号。
- 金额。
- 国家。
- 用途。
- 联系方式。
- 还款计划。
- 状态：pending、approved、rejected。
- 审核备注。
- 审核人。

客服权限建议：

- 客服可查看自己客户贷款申请。
- 管理员负责审核通过/拒绝。
- 如果业务要求客服也能审核，需要单独确认权限。

### 5.6 事件时间线

把关键事件按时间倒序展示：

- 注册。
- 登录。
- 提交卖卡订单。
- 订单状态变化。
- 提现申请。
- 提现完成。
- 贷款申请。
- 贷款审核。
- 视频通话。
- 客服备注变更。

第一期可以先不做完整时间线，只做最近订单/提现/贷款列表。

## 6. 导航调整建议

当前客服登录后只有聊天页，建议调整为：

- 客服登录默认进入 `客服工作台`。
- 底部/侧边导航对客服显示：
  - 工作台
  - 订单
  - 提现
  - 群发
  - 我的

第一阶段可以不单独做所有页面，只先把“工作台”做好：

- 客户列表
- 聊天
- 右侧客户详情

后续再把订单、提现、群发拆成独立大页面。

## 7. 后端接口需求

### 7.1 新增：客服客户详情接口

建议接口：

`GET /api/support/conversations/{conversationId}/customer-profile`

返回：

```json
{
  "conversationId": "support-2",
  "customer": {
    "id": "user-2",
    "username": "alex_crypto",
    "email": "",
    "phone": "",
    "status": "ACTIVE",
    "agentNote": "VIP Apple seller",
    "online": true,
    "assignedAgent": "support_luna",
    "createdAt": "2026-06-24 10:00"
  },
  "balance": {
    "availableTotal": "1,200.00",
    "pendingTotal": "300.00",
    "withdrawnTotal": "500.00"
  },
  "orders": [],
  "withdrawals": [],
  "loans": [],
  "videoSessions": []
}
```

权限：

- AGENT：只能访问自己 assignedAgent 的 conversation。
- ADMIN：可以访问全部 conversation。
- USER：不能访问。

### 7.2 可选：工作台摘要接口

建议接口：

`GET /api/support/workbench/summary`

返回：

- 我的客户数。
- 未读客户数。
- 待处理订单数。
- 待处理提现数。
- 待处理贷款数。
- 今日新增消息数。
- 今日新增订单数。

这个接口不是第一阶段必须，但有助于客服首页顶部统计。

### 7.3 现有接口复用

已有接口可以继续复用：

- `GET /api/support/conversations`
- `POST /api/support/conversations/{conversationId}/note`
- `POST /api/support/conversations/{conversationId}/messages`
- `GET /api/transactions`
- `POST /api/transactions/{transactionId}/status`
- `GET /api/withdrawals`
- `POST /api/withdrawals/{withdrawalId}/status`
- `GET /api/loans`
- `GET /api/balances/summary`
- `GET /api/video-sessions`
- `POST /api/video-sessions`
- `POST /api/broadcasts`

但仅靠这些接口，前端要用用户名过滤各类数据，容易出错。客户详情聚合接口更合理。

## 8. 前端改造需求

### 8.1 类型

新增类型：

- `SupportCustomerProfile`
- `SupportCustomerSummary`
- `CustomerBalanceSummary`

### 8.2 API

新增：

- `fetchSupportCustomerProfile(conversationId)`
- 可选 `fetchSupportWorkbenchSummary()`

### 8.3 Store

新增状态：

- `activeSupportCustomerProfile`
- `supportCustomerProfileLoading`
- `supportCustomerProfileCache`

选择客户时：

1. 切换会话。
2. 清除未读。
3. 拉取客户详情。
4. 展示右栏资料。

### 8.4 页面

改造 `support-chat-v2/index.vue`：

- 外层从双栏改为三栏。
- 左栏保留客户列表。
- 中栏保留聊天。
- 右栏新增客户资料 tabs。

右栏 tabs：

- Overview
- Orders
- Withdrawals
- Loans
- Calls

### 8.5 视觉原则

客服端是运营工具，不要做成营销首页：

- 信息密度要高。
- 列表要易扫读。
- 状态颜色明确但克制。
- 按钮使用清晰动词：处理、完成、争议、查看凭证、备注。
- 卡片不要套卡片，右栏可用分区和表格/紧凑列表。

## 9. 分阶段开发建议

### 阶段 1：客服工作台 MVP

目标：解决“客服只看到聊天，看不到客户业务”的核心问题。

范围：

1. 新增客户详情聚合接口。
2. 客服聊天页改成三栏。
3. 右栏展示客户概览、余额、最近订单、最近提现、最近贷款。
4. 支持订单状态和提现状态快捷处理。
5. 保留现有聊天、备注、翻译、视频、群发能力。

验收：

1. 客服点击客户后，右侧展示该客户资料。
2. 客服能看到该客户订单，不显示其他客户订单。
3. 客服能看到该客户提现，不显示其他客户提现。
4. 客服能看到该客户余额摘要。
5. 客服能在同页回复消息并处理订单/提现。

### 阶段 2：工作台增强

范围：

1. 顶部待办统计。
2. 客户标签/风险标识。
3. 事件时间线。
4. 订单/提现/贷款高级筛选。
5. 管理员全局工作台。

### 阶段 3：独立后台化

范围：

1. 独立客服后台导航。
2. 订单管理大页。
3. 提现管理大页。
4. 客户管理大页。
5. 运营群发管理大页。
6. 数据看板。

## 10. 待确认问题

1. 客服是否可以审核贷款，还是只能查看，由管理员审核？
2. 客服是否可以修改客户状态，例如拉黑/封禁，还是只能管理员操作？
3. 客户余额是否必须精确到单个币种？现在系统金额多为字符串，需要明确币种和计算口径。
4. 订单状态变为 completed 后，是否自动影响用户可用余额？当前余额是按已完成订单减已完成提现计算。
5. 提现提交时是否应该立即扣余额，还是 completed 后扣？二期文档写的是提交后立即扣，但当前实现的汇总口径需要再核对。
6. 客服备注是否只需要一个文本字段，还是需要多标签、风险等级、语言偏好？
7. 管理员是否需要能旁观客服与客户聊天并接管？
8. 客服工作台是否优先支持 PC/H5，移动端只保证可用，还是移动端也要完整三栏体验的替代形态？

## 11. 当前判断

客服只有聊天页是不合理的，尤其在这个项目里，客服承担的是交易跟进和运营处理，而不是单纯聊天。最合理的改法是：保留聊天作为核心，但增加客户业务右栏，把订单、余额、提现、贷款和备注聚合到当前会话上下文里。

建议下一步先做阶段 1。这个阶段改动可控，能最大程度复用现有数据和权限，又能立刻改善客服使用体验。
